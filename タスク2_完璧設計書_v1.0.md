## 2. プロダクト設計の最終形

### 2.1 SDKのアイデンティティ

**Tagline（英）**：
> *The first production-grade TypeScript SDK for GMO Aozora Net Bank's Open API.*

**キャッチコピー（日）**：
> *鹿児島から届ける、sunabar 開発者のための TypeScript SDK*

**Why this SDK?（要約）**：
> sunabar はあるが、TypeScript で本格運用するための基盤が公式には用意されていない。スグクルは特定技能農業派遣の月次給与振込（150件/月）のために自社で全部実装し、それを切り出して OSS にした。

### 2.2 パッケージ構成（最終形）

3パッケージ構成のモノレポ。各パッケージは独立利用可能：

```
@sugukuru/gmo-aozora-sdk          コアSDK（OAuth + API クライアント）
@sugukuru/zengin-format            全銀協ファイル生成（GMO以外でも使える）
@sugukuru/gmo-aozora-webhook       Webhook検証（HMAC-SHA256）
```

**戦略的判断**：`zengin-format` を別パッケージにしたのは、**他銀行ユーザーの取り込み**のため。これだけは GMO 専用ではないので、SMBC・みずほ・地銀ユーザーも依存し始める。コミュニティが拡張する。

## 3. 技術設計の核心

### 3.1 設計原則（コミュニティが信頼する6つの軸）

1. **Type safety first** — Zod v4 で全入出力検証、`any` ゼロ
2. **Production-grade defaults** — PKCE S256 強制、トークン平文保存禁止
3. **Composable, not monolithic** — 3パッケージ独立、他銀行に転用可
4. **No vendor lock-in for the consumer** — Node 20+ 標準 + undici + zod のみ
5. **Honest about limits** — できないことは明示（給与振込専用APIは GMO に存在しない、等）
6. **Battle-tested transparency** — スグクル本番で月150件運用中、と GitHub Issues で実績公開

### 3.2 命名と API 表面

クライアント API はドメイン駆動の自然言語ライク：

```typescript
const client = new GmoAozoraClient({
  environment: "sunabar",
  clientId: process.env.GMO_CLIENT_ID!,
  clientSecret: process.env.GMO_CLIENT_SECRET!,
  redirectUri: "https://app.example.com/callback",
  defaultScopes: [PRIVATE_SCOPES.ACCOUNT, PRIVATE_SCOPES.OFFLINE_ACCESS],
}).useUser("user-123");

// シンプル：1行で取得
const { accounts } = await client.corporation.accounts.list();
const { balances } = await client.corporation.balances.get();

// 自動ページング：イテレータで全件
for await (const tx of client.corporation.transactions.iterate({
  accountId, dateFrom: "2026-04-01", dateTo: "2026-04-30",
})) {
  // ...
}

// バッチ振込 + 結果ポーリング
const bulk = await client.corporation.bulkTransfers.create({...});
const result = await client.corporation.bulkTransfers.pollResult(bulk.requestId, {
  intervalMs: 60_000,
  timeoutMs: 7_200_000,
});

// 全銀協ファイル生成（独立利用可）
import { generateZenginFile, ZENGIN_SHORUI } from "@sugukuru/zengin-format";
const file = generateZenginFile({
  context: {...},
  items: payrollItems,
  shorui: ZENGIN_SHORUI.KYUYO, // 11=給与振込
});
```

### 5.3 production-stories/sugukuru-payroll.md

これが **コミュニティ認知の決定打**。本番事例を生々しく公開する：

```markdown
# Production Story: Sugukuru's Monthly Payroll

## The Setup

Sugukuru Co., Ltd. is a staffing agency in Kagoshima, Japan, dispatching
~150 Indonesian Specified Skilled Workers to agricultural operations across
Kyushu. Every month, we run a payroll batch:

- 150+ recipients
- ¥38M+ total disbursement
- Mix of base salary, overtime, dorm fee deductions
- Compliance check with each worker's residence visa status

## Before (until May 2026)

- Excel spreadsheet → manually formatted Zengin file
- GUI upload to GMO Aozora's online banking
- Phone calls to fix misformatted records
- ~14 hours/month of pure transcription work
- One typo in 2025: caused a ¥240,000 misroute that took 3 days to recover

## After (from June 2026, with this SDK)

```ts
// 1. Generate Zengin file (compile-time validation of shorui code)
const file = generateZenginFile({...});

// 2. Submit via bulk transfer API (JSON, not file upload)
const bulk = await client.corporation.bulkTransfers.create({...});

// 3. Poll for results
const result = await client.corporation.bulkTransfers.pollResult(bulk.requestId);
```

That's it. Compile-time validation prevents the kind of typo that bit us before.

## Numbers

| Metric | Before | After |
|---|---|---|
| Manual processing time | 14h/month | 1.5h/month |
| Typo-related incidents | 1-2/year | 0 since June 2026 |
| Time from "approval" to "all paid" | 4 hours | 8 minutes |
```

これは**「単なる SDK 紹介」ではなく、地方の中小企業が銀行APIで救われた事例**として、sunabar コミュニティの心に刺さる構造。

## 7. リスクとミティゲーション

### 7.1 想定リスク

| リスク | 確率 | インパクト | ミティゲーション |
|---|---|---|---|
| **GMO 仕様変更**で SDK が壊れる | 中 | 高 | バージョン固定、CHANGELOG 厳格管理、複数銀行抽象化レイヤ準備 |
| **コミュニティ無反応**（Star も DL も伸びない）| 中 | 高 | Zenn 連載+寄稿で見える化、自社実例を最大の素材に |
| **競合 SDK 出現** | 低 | 中 | 1.0 リリース速度勝負、コミュニティへの先行者優位を確立 |
| **GMOから「公認」されない** | 中 | 中 | 公認なしでも OSS 自体の価値で勝負、別ルートで認知獲得 |
| **時間が取れない**（壁さんの本業逼迫）| 高 | 高 | Cursor 自動化で実装時間 90% 削減、Phase 区切りで進行 |
| **セキュリティ脆弱性発覚** | 低 | 致命的 | dependabot、CodeQL、初版から security audit、SECURITY.md 整備 |
| **GMO担当者経由の橋渡しが断られる** | 低 | 中 | 公式問い合わせフォーム、Twitter経由など複数ルート確保 |
| **登壇に至らない** | 中 | 中 | 登壇は最終目標であって唯一ではない、認知 Lv2 で十分実用価値 |

### 7.2 失敗時の最低ライン

最悪ケース（コミュニティ無反応、登壇なし）でも：

1. **SDK は実用品**（スグクル本業で月150件の給与振込が回る）
2. **OSS 公開実績**（壁さんの個人ブランド資産）
3. **タスク1の準備完了**（sugukuru-finance MCP の銀行レイヤーが完成済み）
4. **ドメイン知識の整理完了**（後で SSW Compass リブランドの基礎）

→ **失敗しても Lose しない構造**。これが本書の戦略の本質。
